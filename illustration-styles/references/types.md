# Types

The 6 ways an illustration can structure information. Pick the Type that matches what the article is *doing* in the place where the image will sit — not the topic of the article overall.

| Type | Best for | Avoid for |
|------|----------|-----------|
| `infographic` | Data, metrics, technical breakdowns, "anatomy of X" | Pure narrative, emotional moments |
| `scene` | Narrative, emotional context, lifestyle, moments | Hard data, comparisons |
| `flowchart` | Step-by-step processes, workflows, onboarding | Standalone concepts, single ideas |
| `comparison` | Side-by-side options, before/after, A vs B | Solo concepts, sequential steps |
| `framework` | Models, architectures, role/component relationships | Time-based change, emotional content |
| `timeline` | History, milestones, evolution, growth narratives | Snapshots, single moments |

## Picking The Type

Ask: *what shape does the reader's understanding take after they look at this image?*

- A grid of facts → `infographic`
- A felt scene → `scene`
- A path A→B→C → `flowchart`
- Two columns that contrast → `comparison`
- A system you can name the parts of → `framework`
- A line moving left-to-right through time → `timeline`

When the same article needs different shapes at different points, use different Types for different illustrations. Don't force one Type across a whole article just for consistency — that produces a manual, not an article.

## Type Goes With Style And Palette

Type alone doesn't determine the look. Type sets the *information structure*; Style sets the *visual character*; Palette sets the *color discipline*. See `compatibility.md` for which Style works with which Type.
